Commitment Issues

Why the content of your commit messages matters more than you might think.

Commit messages can often seem like a bit of an annoyance. You can view the code changes you made easily and, so it can often feel like you’re repeating yourself or entering largely redundant text, with messages like “Corrected a typo”, “Renamed a function” or “Made the thing work that wasn’t working”.

If you’re working in a company rather than contributing to something open-source you probably don’t feel the need to use commit messages to explain your changes — that’s handled by some sort of issue tracking software.

So why bother?

Well, recently I found myself trying to understand a change that I needed to make to some software. It involved changing a database query so that it worked with some new columns that were introduced in a previous new feature. This is a product that does little more than query one database and make inserts into another, for various complicated reasons.

The ticket was very clear, stating the columns that I should be querying and how they mapped to the target database.

But one of the columns I needed to query simply didn’t exist. I searched around in our issue tracking software to find out whether it had been added as part of the earlier feature, as the ticket seemed to imply. I couldn’t find any mention of database changes at all, so it seems they’d been left to the developer to decide on. Worse still, that developer had left the company the week before.

So, I turned to git. I would find the commits related to that feature and see if the missing column had been introduced then, and subsequently removed, or if it never existed. That would give me a clear question I could ask of the product owner (“This was never there, should it have been?” or “This was removed in feature X, should I put it back?” or “This was renamed…” you get the idea.)
